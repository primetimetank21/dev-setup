### 🔍 Root Cause Notes

**Bug #1 — `$MyInvocation.MyCommand.Path` null in hosted contexts:**
- Root cause: `$MyInvocation.MyCommand.Path` is only populated when a script is invoked directly as a file. In dot-sourced, piped, or hosted environments it returns `$null`. `$PSScriptRoot` has been the correct answer since PS 3.
- Pattern: Any PS script using `$MyInvocation.MyCommand.Path` for self-location is broken in hosted contexts. Should be caught in code review.
- Fix rule: **Always use `$PSScriptRoot` for script-relative paths in `.ps1` files.** Documented and merged.

**Bug #2 — `$IsLinux`/`$IsMacOS`/`$IsWindows` undefined on PS 5.x:**
- Root cause: These variables were introduced in PowerShell 6 (Core). Under `Set-StrictMode -Version Latest`, referencing an undefined variable is a hard error. `setup.ps1` sets strict mode and used these variables without version-guarding.
- Pattern: **PS 6+ automatic variables must always be guarded behind a `$PSVersionTable.PSVersion.Major -ge 6` short-circuit check.** The `-and` short-circuit means the undefined variable is never evaluated on PS 5.x.
- Underlying pattern: The codebase has no PS 5.x compatibility checklist. Both bugs were introduced in the same file, in the same function — they share a root: Windows platform code was written and tested only on PS 7+.

**Common thread:** Windows PowerShell 5.x is still the default on stock Windows 10/11 installs. If we can't run CI on it and don't have a PS 5.x compat checklist, bugs like these will recur every time someone writes new Windows code without actively testing PS 5.1.
